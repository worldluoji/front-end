### 场景模拟

```javascript
async function f() {
  await fetch('/api/save');          // ① 异步操作
  location.href = '/new-page';      // ② 跳转（在 f 的 await 恢复后执行）
}

async function a() {
  await f();                         // ③ 等待 f() 完成
  navigator.sendBeacon('/log', data); // ④ 埋点（永远不可靠）
}
```

**执行时序与事件循环：**

1. `a()` 执行到 `await f()`，调用 `f()`。  
   `f` 内部执行到 `await fetch(...)`，返回一个 Promise，并交出控制权。
2. 主线程继续其他任务。当 `fetch` 完成，`f()` 里的代码作为**微任务**恢复执行。
3. 微任务执行 `location.href = '/new-page'`——**跳转指令发出**。
4. `f()` 的 Promise 变为 resolved，`a()` 里 `await f()` 后面的代码（第④步）作为一个**新的微任务**排入微任务队列。
5. 此时微任务队列尚未清空，但浏览器接收到导航指令后，会在当前任务结束后准备跳转。不同的浏览器行为可能略有差异，但普遍会**在页面卸载前清空微任务队列的可能性极低**——因为导航启动通常比微任务更优先，且在新页面开始加载前，旧页面的上下文可能随时被销毁。
6. 即便第④步微任务侥幸开始执行，`sendBeacon` 发出的请求也可能因页面卸载而被中断（即使是 `sendBeacon` 也只能保证在 `unload` 等可靠卸载事件中发送，此时并不在那些事件里）。

**结论：** 第④步埋点要么完全不执行，要么发送一个会被取消的请求，**不能用于实际埋点**。

---

### 为什么它“几乎一定不会执行”？

- **跳转的同步性被 async 打破**：在同步代码中直接写 `location.href = ...` 时，后面的同步代码仍会执行完再跳转，微任务也有机会执行。但 `async f` 把跳转放到了 `await` 之后，它本身就已经是一个微任务，恢复后发出的跳转会在当前任务结束时触发导航，导致队列中后续的微任务没有执行窗口。
- **浏览器导航的时机**：导航通常在任务（task）结束时发生，微任务队列清空是在任务内部完成的，但导航请求可能打断该过程，特别是当跳转目标不同域或需要卸载页面时。

---

### 最佳实践

**坚决不要在跳转之后的任何异步操作里做关键逻辑**（埋点、状态同步等）。必须**先完成所有关键操作，再跳转**：

```javascript
async function f() {
  await fetch('/api/save');
  // ✅ 先埋点（或在 f 外部 a 中保证在跳转前执行）
  navigator.sendBeacon('/log', data);
  location.href = '/new-page';
}
```

或者把跳转和业务逻辑解耦，让调用方控制跳转时机：

```javascript
async function f() {
  await fetch('/api/save');
  // 不做跳转，只返回结果
}

async function a() {
  await f();
  // ✅ 在此处上报，因为跳转还没发生
  navigator.sendBeacon('/log', data);
  location.href = '/new-page;
}
```

如果确实需要在跳转前保证上报，`navigator.sendBeacon` 是最佳选择，但也必须在跳转指令前调用。不要试图在跳转之后的 `await` 恢复里兜底，那里已经太晚了。
